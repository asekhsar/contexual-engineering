Contextual Engineering in Agentic Workflow

I wanted to explain contextual engineering through something concrete, so I built the narrative around a multi-agent travel assistant. The system does not feel personalized because one prompt is clever. It feels personalized because the workflow knows what to remember, what to update, what to route forward, and what to leave out. In this post, I use that travel assistant to explain three ideas that matter far more in production than in toy demos: state management, user feedback accomplishment, and dynamic context through static and dynamic instructions.

The travel assistant begins by identifying the user, loading long-term preferences, and collecting trip intent. It then moves through flight booking, hotel booking, itinerary creation, and itinerary refinement. At each stage, the manager agent decides which specialist should run next and which slice of context belongs in that call. That is the core of contextual engineering: the system behaves intelligently because context is curated step by step rather than dumped wholesale into one giant prompt.

Context engineering is really a systems problem: what the model sees, what it remembers, what it retrieves just in time, and what it hands off after each stage.

The useful question is not only “what prompt should I write?”

It is also “what should this agent know right now, what should it ignore, and what should be handed off next?”

From prompt engineering to contextual engineering

Prompt engineering matters, but it is a narrower discipline. It optimizes one interaction. Contextual engineering is the more mature systems view: it asks what the model should see, what memory should be loaded, what state should be trusted, what fresh results belong in the next call, and how that context should change after each output.

In this travel system, the planner, booking agents, and itinerary agents behave coherently not because they all see everything, but because the manager keeps curating the right combination of stable rules, persistent memory, current task state, and the latest user feedback before every handoff.

How the travel assistant works

The root manager orchestrates the workflow. The travel planner establishes identity and trip intent. The flight booking agent handles timing, airport logic, and airline preferences. The hotel booking agent uses approved flight dates, family composition, and budget cues to find the right stay. The itinerary agent then builds a plan using all approved context, and the itinerary refiner loops on user feedback until the result is accepted.

That sounds simple in prose, but under the hood it only works because the system is stateful. User identity and preferences persist across sessions. Pending selections live in short-term memory while the workflow is active. Approvals convert temporary state into confirmed state. And every specialist agent gets a narrow context slice rather than the entire conversation history.

Why this is a good way to explain contextual engineering

Travel planning naturally forces the workflow to be incremental. Dates matter. Family composition matters. Dietary restrictions matter. Weekend flexibility matters. The hotel decision changes what makes sense for the itinerary. Feedback changes the itinerary again. It is exactly the kind of environment where poor context management leads to brittle behavior, repeated questions, and generic outputs.

That is why this case study is useful: the system has to preserve continuity without overloading the model, and it has to route the right context to the right specialist at the right time. Anthropic’s context-engineering guidance frames the core challenge in almost the same way: context is finite, iterative, and needs to be curated at every step of the agent loop.[1]

A manager-orchestrated architecture with scoped specialists

I did not want one overloaded generalist prompt. I wanted a manager that routes control across specialist agents. The planner is responsible for user identity and trip intent. The booking agents focus on flights and hotels. The itinerary pair handles planning and revision. This makes the workflow easier to reason about and much easier to control from a context perspective.

State management is the real operating layer

The most important part of this system is not the prose in the prompts. It is the state model. I persist stable user preferences across sessions, and I keep short-term task state alive across the current workflow. Long-term memory tells the system who the user is and how they usually travel. Short-term memory tells the workflow what is currently pending, what has already been approved, and what should happen next.

Feedback is Not Just Input; It’s a State Transition

Explicit feedback loops allow for human-in-the-loop control, increasing trust and reliability. Feedback is not just more text; it is a signal used to refine state until an explicit approval is received. This idea is strongly aligned with Anthropic’s emphasis on iterative context curation and with ADK-style production agent design where state and handoffs are first-class concerns.[1] [2]

Dynamic context means combining stable rules with live state

Every agent in this workflow runs with static instructions and dynamic context. Static instructions define the role, the output format, the stage boundary, and the things the agent must not assume. Dynamic context carries the current user state, approved outputs from earlier stages, latest feedback, and active task-specific details. That layering is what makes specialist agents feel informed without becoming overloaded.

Static instructions

  • Who the agent is
  • What the agent is responsible for
  • Which tools or actions it may use
  • What it must not assume
  • How it should format the next output

Dynamic context

  • Returning user profile and saved preferences
  • Approved trip details
  • Selected flight and hotel
  • Fresh search outputs
  • The latest correction, revision, or approval signal

What this looks like in practice

The planner should know identity, prior preferences, and trip intent. The flight agent should know home city, traveler count, flexibility, and flight dislikes. The hotel agent should know approved dates, family composition, amenity needs, and hotel budget. The itinerary agent should know what has already been approved plus dietary restrictions, pace, and the latest feedback cycle.

Why this matters technically

When each agent gets only the context that belongs to its stage, context windows stay leaner and the model is less likely to drift into irrelevant reasoning. Anthropic describes effective context engineering as continuously curating a finite context resource; the ADK-oriented production view similarly emphasizes shaping state, instructions, and interfaces around each step.[1] [2] [3]
